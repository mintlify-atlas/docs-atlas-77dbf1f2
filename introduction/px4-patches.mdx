PX4 maintains a fork of Apache NuttX at [github.com/PX4/NuttX](https://github.com/PX4/NuttX). The fork tracks the upstream Apache NuttX project closely and contributes non-PX4-specific improvements back upstream. On top of that shared foundation, PX4's fork adds board support packages (BSPs) for the MCU families used in Pixhawk-series flight controllers, a set of GitHub Actions CI workflows, and Kconfig configuration defaults tuned for autopilot workloads.

This page documents what is present in the PX4 fork that is not part of the upstream NuttX tree.

## CI workflows

The `.github/workflows/` directory contains five GitHub Actions workflow files that run on every pull request and push to `master` or release branches:

| Workflow file | Purpose |
|---|---|
| `build.yml` | Compiles NuttX for a matrix of board targets to catch build regressions. Runs on `pull_request` and pushes to `master` and `releases/*`. |
| `check.yml` | Runs static analysis and Kconfig consistency checks. |
| `lint.yml` | Enforces the NuttX C coding standard using the `nxstyle` tool. |
| `doc.yml` | Builds the Sphinx documentation to verify it renders without errors. |
| `docker_linux.yml` | Builds and pushes the Docker image used as the CI build environment. |

The `dependabot.yml` file configures Dependabot to keep GitHub Actions dependencies up to date. The `.github/linters/` and `.github/actions/` directories contain supporting configuration and composite actions used by the workflows above.

<Info>
  Pull requests to the PX4 NuttX fork must pass all five workflow checks before merging. Style errors caught by `lint.yml` must be corrected even if the contributor did not introduce them — each PR is expected to leave the modified files compliant with the NuttX C coding standard.
</Info>

## Board support package structure

Each board directory follows the standard NuttX layout:

```
boards/arm/<chip>/<board>/
├── configs/
│   └── <config-name>/
│       ├── defconfig        # Compressed Kconfig configuration
│       └── Make.defs        # Toolchain and linker settings (optional)
├── scripts/
│   ├── Make.defs            # Shared toolchain settings for the board
│   └── <board>.ld           # Linker script
├── src/
│   ├── Makefile
│   ├── <board>_boot.c       # Early board initialisation
│   ├── <board>_bringup.c    # Driver and peripheral initialisation
│   └── <board>_userleds.c   # LED driver (where present)
└── README.txt               # Board-specific notes
```

To instantiate a configuration, use:

```bash
tools/configure.sh <board>:<config-name>
make olddefconfig
```

## PX4-specific Kconfig options

PX4 board configurations commonly set the following Kconfig options that differ from upstream NuttX defaults:

- **`CONFIG_SCHED_HPWORK`** and **`CONFIG_SCHED_LPWORK`** — high- and low-priority work queues required by the PX4 middleware layer.
- **`CONFIG_BOARDCTL`** and related `CONFIG_BOARDCTL_*` options — board control interface used by the PX4 startup sequence.
- **`CONFIG_FS_PROCFS`** — procfs enabled for runtime diagnostics and parameter access.
- **`CONFIG_SERIAL_TERMIOS`** — TERMIOS support required by MAVLink serial transports.
- **`CONFIG_USBDEV`** and **`CONFIG_CDCACM`** — USB CDC/ACM serial used for USB console and MAVLink.

<Tip>
  When porting NuttX to a new Pixhawk board, start from the nearest existing board's `defconfig` and run `make menuconfig` to adjust peripheral assignments. Use `make savedefconfig` to write the resulting compressed `defconfig` back to the board directory.
</Tip>
